# 분석 및 워크샵 운영 프로토콜 v2

- 최초 작성일자: 2026-05-12
- 업데이트일자: 2026-05-12
- 업데이트 내용: v1.3 팀원 중심 응답 대상, 제품별 책임 역할표, 어려운 용어 정리 반영.

## 0. 문서 상태

이 문서는 설문 결과 분석과 워크샵 운영 기준이다. 실제 설문 문항 구현 기준은 `02_survey_design/07_final_question_bank_source_of_truth.md`를 따른다.

## 1. 이 문서의 목적

설문은 수집 자체보다 해석과 후속 실행이 중요하다. 이 문서는 설문 결과를 어떻게 읽고, 워크샵에서 어떤 논의로 바꾸며, 4주 실험으로 어떻게 연결할지 정의한다.

## 2. 분석 원칙

### 2.1 개인 평가 금지

분석 단위는 개인이 아니라 운영 시스템이다.

금지:

- 특정 개인/직군을 문제 원인으로 지목
- 소수 직군 평균 공개
- 개인이 식별되는 주관식 직접 인용
- 팀장/리드/PM/엔지니어 개인 평가처럼 해석

허용:

- 의사결정 구조 병목
- 제품 우선순위 병목
- 제품별 실행 기준과 공유 체계 부족
- 외부 협업 인터페이스 문제
- 온보딩/문서화 부족
- 동시에 진행 중인 일과 업무 전환 과다

### 2.2 평균, 분산, 하위 점수 비율을 함께 본다

각 문항/축은 다음 값으로 본다.

- 평균
- 1~2점 비율
- 4~5점 비율
- 표준편차 또는 응답 분산
- 주관식 반복 키워드

해석 기준:

- 평균 4.2 이상: 강점 후보
- 평균 3.6~4.1: 작동 중이나 개선 여지 있음
- 평균 3.0~3.5: 불안정 영역
- 평균 2.9 이하: 구조적 개입 필요
- 1~2점 비율 30% 이상: 평균과 무관하게 워크샵 논의 필요
- 분산이 큰 문항: 제품/역할/신규 여부에 따른 경험 차이 가능성

### 2.3 주관식은 빈도와 강도를 나눠 본다

빈도가 높은 문제만 중요한 것은 아니다. 소수 응답이라도 다음 조건이면 중요하게 다룬다.

- 고객 신뢰 리스크
- 보안/개인정보/저작권 리스크
- 장애/운영 리스크
- 번아웃 리스크
- 제품 방향성 혼선
- 반복 재작업을 유발하는 구조 문제

## 3. 권장 분석 산출물

### 3.1 1페이지 요약

워크샵 시작 전 공유할 요약은 다음 구조가 좋다.

1. 전체 응답 현황
2. 상위 강점 3개
3. 하위 개선 영역 3개
4. 분산이 큰 문항 3개
5. 주관식에서 반복된 병목 5개
6. 워크샵에서 반드시 결정해야 할 질문 3개

### 3.2 9개 진단 축별 상세표

각 축마다 다음을 정리한다.

| 축 | 평균 | 1~2점 비율 | 4~5점 비율 | 해석 | 워크샵 질문 |
|---|---:|---:|---:|---|---|
| 사업/제품 정렬 | X.X | X% | X% | 제품 우선순위 정렬 수준 | 어떤 제품 기준이 가장 모호한가 |

### 3.3 병목 지도

설문 결과는 다음 5개 병목 유형으로 재분류한다.

1. 전략 병목: 무엇을 왜 우선하는지 불명확
2. 고객 신호 병목: 고객/사용자/내부 피드백이 제품 판단으로 연결되지 않음
3. 협업 인터페이스 병목: 팀 내부 제품/개발 협업 또는 타팀 협업의 연결이 약함
4. 제품 실행 병목: 운영 제품과 CBT/실험 제품의 실행 기준, 공유 수준, 제품화 기준이 모호함
5. 실행 용량 병목: 동시에 진행 중인 일, 회의, 의사결정 대기, 업무 전환 과다

## 4. 워크샵 운영안

### 4.1 사전 준비

워크샵 전 준비물:

- 설문 요약 1페이지
- 9개 진단 축 상세표
- 주관식 테마 요약
- 제품/업무 영역별 현재 주요 업무 목록
- 외부 협업 접점 목록
- 제품별 책임 역할표 초안

사전 공유 시 주의:

- 원문 응답 전체를 공유하지 않는다.
- 개인 식별 가능 표현은 제거한다.
- 낮은 점수를 비난 자료처럼 보이게 하지 않는다.

### 4.2 세션 1: 전체 결과 확인

목표:

- 모두가 같은 데이터에서 출발한다.
- 좋은 점과 문제를 균형 있게 본다.

진행:

1. 진행자가 조사 목적과 익명성 원칙을 다시 말한다.
2. 상위 강점 3개를 먼저 본다.
3. 하위 개선 영역 3개를 본다.
4. 분산이 큰 문항을 본다.
5. 해석이 필요한 질문을 적는다.

금지:

- "누가 이렇게 답했을까" 추정
- 특정 직군 방어
- 즉시 해결책으로 뛰어가기

### 4.3 세션 2: 병목 분류

목표:

- 문제를 사람 문제가 아니라 운영 병목으로 바꾼다.

방법:

각자 가장 중요한 병목 3개를 고른 뒤, 아래 5개 유형에 붙인다.

- 전략 병목
- 고객 신호 병목
- 협업 인터페이스 병목
- 제품 실행 병목
- 실행 용량 병목

논의 질문:

- 이 병목은 어느 제품에서 가장 크게 나타나는가?
- 이 병목은 반복적인가, 일회성인가?
- 이 병목을 방치하면 어떤 고객/품질/일정 리스크가 생기는가?
- 이 병목은 팀 내부에서 해결 가능한가, 외부 협업이 필요한가?

### 4.4 세션 3: 분과별 논의

14명 팀원이 한 번에 논의하면 발언 기회가 불균형해질 수 있다. 주제는 5개 분과로 유지하되, 실제 방 구성은 참석자 수와 선택 분과에 따라 3~5개로 조정한다.

기준 분과:

1. 제품/우선순위 분과
2. 고객/시장/내부 피드백 분과
3. PM/프롬프트/요구사항 분과
4. 엔지니어링/운영 분과
5. CBT/신규 실험 분과

인원이 적으면 다음처럼 합칠 수 있다.

| 실제 방 | 포함 주제 |
|---|---|
| 제품/고객 방 | 제품/우선순위, 고객/시장/내부 피드백 |
| 요구사항/실험 방 | PM/프롬프트/요구사항, CBT/신규 실험 |
| 엔지니어링/운영 방 | 엔지니어링/운영, 외부 협업 병목 |

각 그룹 산출물:

- 가장 중요한 병목 1개
- 현재 증상
- 가능한 원인
- 4주 안에 바꿀 수 있는 작은 실험
- 실험 성공 기준
- 필요한 의사결정자 또는 협업팀

### 4.5 세션 4: 4주 실험 결정

워크샵의 최종 산출물은 큰 조직개편안이 아니라 실행 가능한 4주 실험이어야 한다.

실험 템플릿:

| 항목 | 내용 |
|---|---|
| 실험명 | 짧고 구체적으로 |
| 해결하려는 병목 | 설문 문항/주관식 근거 연결 |
| 대상 제품/업무 영역 | Agent, Viewer, Reader, Idea, Editor, 공통 플랫폼 등 |
| 변경할 운영 방식 | 회의, 문서, 의사결정, 평가, 배포 등 |
| 책임 역할 | 개인명보다 역할을 먼저 정의한 뒤 내부에서 지정 |
| 참여자 | 필요한 직군/협업팀 |
| 성공 기준 | 4주 뒤 확인 가능한 지표 |
| 리뷰 날짜 | 워크샵 당일 확정 |

## 5. 추천 4주 실험 후보

설문 결과를 보기 전의 후보이므로 확정안이 아니다. 결과에 따라 선택한다.

### 실험 1. 제품별 책임 역할과 우선순위 원칙 1페이지화

문제 가설:

- 여러 제품이 동시에 움직이며 책임 범위와 우선순위가 흐려질 수 있다.

실험:

- Agent, Viewer, Idea, Reader, Editor, 공통 플랫폼 각각에 대해 책임 역할, 현재 목표, 이번 달 하지 않을 일을 1페이지로 정리한다.

성공 기준:

- 4주 뒤 Q2, Q5, Q19 관련 체감 개선.

### 실험 2. 운영 제품과 CBT/실험 제품 기준 정리

문제 가설:

- 운영 중인 제품과 CBT/실험 단계 제품의 공유, 검증, 의사결정 기준이 섞이면 속도와 정렬이 모두 떨어질 수 있다.

실험:

- 운영 제품과 CBT/실험 제품을 구분해 공유 주기, 검증 수준, 제품화 기준, 리스크 체크 항목을 1페이지로 정리한다.

성공 기준:

- AI Editor 등 CBT/실험 업무 영역에 적용.
- 4주 뒤 제품별 공유 기준과 제품화 기준에 대한 혼선 감소.

### 실험 3. 요구사항 템플릿 개편

문제 가설:

- AI 기능은 일반 기능보다 불확실성이 큰데, 요구사항이 이를 충분히 반영하지 못할 수 있다.

실험:

- 신규 기능 티켓에 사용자 문제, 대상 사용자, 성공 기준, 기술 제약, UX 한계 안내, 운영 영향 항목을 추가한다.

성공 기준:

- 신규 기능 2개 이상에 적용.
- 재작업/질문 감소 체감.

### 실험 4. 외부 협업 접점 정례화

문제 가설:

- 인프라, 디자인, 영업/서비스/콘텐츠 피드백이 늦게 들어와 일정과 품질에 영향을 줄 수 있다.

실험:

- 외부 협업 접점별로 필요한 입력, 요청 방식, 응답 기대 시간을 정리한다.

성공 기준:

- 가장 큰 외부 병목 1개에 대해 처리 방식 합의.

### 실험 5. 신규/확장 인원 온보딩 맵

문제 가설:

- 신규 구성원이 도메인/시스템/의사결정 맥락을 파악하는 데 암묵지 의존이 크다.

실험:

- DBpia AI 제품 구조, 데이터 소스, 검색/RAG 흐름, 주요 협업팀, 자주 묻는 질문을 1개 문서로 묶는다.

성공 기준:

- 신규/기존 구성원 모두가 "어디서 찾아볼지" 알게 됨.

## 6. AI 분석 프롬프트에 반영할 원칙

설문 앱의 AI 분석 프롬프트는 기존 4Q 맥락을 반드시 교체해야 한다.

새 분석 프롬프트 핵심 배경:

1. 누리미디어 AI 사업부는 학술 AI 제품을 만드는 16명 조직이며, 실무 스쿼드는 15명이다. 이번 설문 분석의 기본 응답 대상은 팀장과 스쿼드 리드를 제외한 실무 팀원 14명이다.
2. 담당 제품은 AI Agent, AI Viewer, AI Idea, AI Reader, AI Editor다.
3. 비즈니스는 B2B2C이며 대학교 도서관이 핵심 고객이다.
4. 실제 사용자는 학생/연구자이고, 내부적으로 영업/마케팅/서비스/콘텐츠/인프라와 협업한다.
5. 분석 목적은 개인 평가가 아니라 운영체계 병목과 4주 실험 도출이다.
6. AI/RAG 품질, 근거성, 검색 커버리지, 잘못된 답변 위험, 비용/응답 속도 사이의 균형은 공통 조직 설문 핵심 축이 아니라 필요 시 엔지니어링/운영 분과의 선택 토픽으로 본다.

AI 분석 시 금지:

- 근거 없이 조직문화 문제로 일반화
- 특정 개인/직군 탓으로 해석
- 낮은 점수를 바로 리더십 실패로 단정
- 소수 응답의 개인 식별 가능 인용
- "스타트업처럼 더 빠르게" 같은 추상적 권고

권장 출력:

- 강점 3개
- 병목 5개
- 제품/고객/협업/제품별 실행 방식/실행용량별 해석
- 워크샵 토론 질문
- 4주 실험 후보

## 7. 최종 산출물 체크리스트

워크샵이 끝났을 때 최소한 다음은 남아야 한다.

- 제품별 현재 우선순위와 책임 범위에 대한 합의
- 제품별 책임 역할표 업데이트본
- 운영 제품과 CBT/실험 제품의 실행 기준에 대한 합의
- 외부 협업 병목 1개에 대한 조치
- 온보딩/문서화 개선 항목 1개
- 4주 뒤 리뷰할 운영 실험 2~3개
- 월간 펄스에서 확인할 문항 후보 5개 이하

## 8. 분과별 토론 모듈 연결

분과별 토론의 상세 문항과 입력 양식은 `02_survey_design/06_breakout_discussion_modules_v2.md`를 기준으로 한다.

## 9. 월간 펄스 연결

워크샵에서 4주 실험 2~3개를 선택한 뒤, `02_survey_design/08_monthly_pulse_followup_pool.md`에서 실험과 연결되는 5문항 이하를 고른다.

펄스 해석 원칙:

- 점수는 실험의 성패를 단정하는 자료가 아니라 조정 신호로 본다.
- N/A 비율이 높으면 실험 공유 또는 적용 범위 문제로 해석한다.
- 주관식 조정 의견이 반복되면 다음 달 실험 조건을 바꾼다.
