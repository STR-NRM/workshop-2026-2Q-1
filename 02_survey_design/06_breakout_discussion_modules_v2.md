# 분과별 토론 모듈 v2

- 최초 작성일자: 2026-05-12
- 업데이트일자: 2026-05-13
- 업데이트 내용: v1.3 사용자 화면 문구 전수 점검 결과 반영. 토론 입력 문항도 쉬운 한국어 기준으로 정리.

## 0. 문서 상태

이 문서는 워크샵 분과 토론 운영 기준이다. 개인 설문 문항의 최종 구현 기준은 `02_survey_design/07_final_question_bank_source_of_truth.md`를 따른다.

## 1. 목적

개인 설문은 각자의 체감과 일이 막히는 지점을 수집하기 위한 도구다. 분과별 토론은 개인 응답을 바탕으로 "팀이 실제로 바꿀 수 있는 작은 개선"을 합의하기 위한 단계다.

이 문서는 다음 요구를 반영한다.

1. 모두가 개인 설문을 먼저 수행한다.
2. 이후 분과별로 모여 토론한다.
3. 각 분과는 토론 결과를 별도 입력한다.
4. 분과 입력은 개인 의견이 아니라 그룹 합의 또는 주요 의견 차이를 기록한다.
5. AI/RAG 신뢰성처럼 특정 직군 중심 주제는 공통 설문이 아니라 관련 분과에서 다룬다.

## 2. 운영 흐름

### 2.1 1단계: 개인 설문

모든 구성원이 공통 설문에 응답한다.

구성:

- 기본 정보
- 공통 조직 운영 문항
- 직무별 추가 문항
- 개인 서술형 문항

### 2.2 2단계: 제품별로 끝까지 챙기는 역할표 사전 작성

분과 토론 전에 제품/업무 영역별 현재 책임 범위를 정리한다. 이 표는 개인 평가가 아니라 결정 방식과 책임 흐름을 명확히 하기 위한 도구다.

| 제품/업무 영역 | 끝까지 책임지고 챙기는 역할 | 부재 시 대신 챙길 역할 | 주요 참여 역할 | 의견 확인/공유 대상 | 이번 반기 하지 않을 일 |
|---|---|---|---|---|---|
| AI Agent |  |  |  |  |  |
| AI Viewer |  |  |  |  |  |
| AI Idea |  |  |  |  |  |
| AI Reader |  |  |  |  |  |
| AI Editor |  |  |  |  |  |
| 공통 검색/AI 플랫폼/인프라 |  |  |  |  |  |

작성 원칙:

- 개인 이름을 공개 토론에서 평가하지 않는다.
- 비어 있는 칸은 문제라기보다 확인이 필요한 영역으로 본다.
- 끝까지 책임지고 챙기는 역할이 여러 명일 수는 있지만, 최종 결정을 마무리하는 역할은 별도로 확인한다.
- "이번 반기 하지 않을 일"은 동시에 진행 중인 일을 줄이기 위한 항목이다.

### 2.3 3단계: 결과 요약 공유

진행자는 개인 설문 결과를 요약해 공유한다.

공유 가능:

- 전체 평균
- 하위/상위 영역
- 응답 차이가 큰 문항
- 반복 서술형 테마
- 일이 막히는 지점 후보

공유 금지:

- 개인 식별 가능한 원문
- n이 작은 직군별 평균
- 특정 개인/팀을 암시하는 표현

### 2.4 4단계: 분과별 토론

권장 분과는 다음 5개다.

1. 제품/우선순위 분과
2. 고객/시장/관련 팀 의견 분과
3. PM/프롬프트/요구사항 분과
4. 엔지니어링/운영 분과
5. CBT(사전 테스트)/신규 실험 분과

AI 품질/RAG 세부 이슈는 필요할 때 엔지니어링/운영 분과의 하위 토픽으로 다룬다.

### 2.5 5단계: 분과 결과 입력

각 분과는 토론 후 동일한 형식으로 결과를 입력한다.

필수 입력:

- 가장 중요한 막히는 지점 1~2개
- 막힘의 근거
- 4주 안에 바꿀 수 있는 작은 개선
- 개선을 끝까지 챙기는 역할
- 필요한 협업 대상
- 성공 기준
- 남은 의견 차이

## 3. 공통 분과 입력 양식

### G1. 이 분과에서 가장 중요하다고 본 막히는 지점은 무엇입니까?

설명: 개인 문제가 아니라 일하는 방식, 결정 방식, 협업 구조, 정보 흐름 관점으로 적어주세요.

### G2. 그 막힘이 실제로 존재한다고 본 근거는 무엇입니까?

설명: 설문 결과, 반복 경험, 최근 사례, 데이터 중 가능한 근거를 적어주세요. 개인 식별 가능 표현은 제외합니다.

### G3. 이 막힘을 방치하면 어떤 위험이 가장 큽니까?

선택지:

- 제품 우선순위 혼선
- 고객/사용자 가치 저하
- 일정 지연
- 재작업 증가
- 품질/운영 위험
- 구성원 피로도 증가
- 타팀 협업 비용 증가
- 기타

### G4. 다음 4주 동안 실행할 수 있는 작은 개선은 무엇입니까?

설명: 조직개편이나 큰 시스템 구축보다, 4주 안에 시작하고 확인할 수 있는 운영 변경을 적어주세요.

### G5. 개선 성공 여부는 무엇으로 확인합니까?

설명: 숫자 지표가 아니어도 됩니다. 예: 결정 대기 감소, 재작업 감소, 회의 시간 감소, 문의 경로 명확화, 템플릿 적용 건수 등.

### G6. 이 분과 안에서 합의되지 않은 의견 차이가 있다면 무엇입니까?

설명: 합의되지 않은 의견도 중요한 신호입니다. 다수 의견과 소수 의견을 구분해 적어주세요.

## 4. 분과별 토론 모듈

### 4.1 제품/우선순위 분과

대상:

- PM, 제품 결정에 자주 관여하는 구성원
- 단, 원하는 구성원은 참여 가능

핵심 질문:

1. Agent, Viewer, Idea, Reader, Editor의 현재 우선순위는 명확한가?
2. 운영 제품과 CBT(사전 테스트)/실험 제품의 판단 기준은 달라야 하는가?
3. 이번 반기에 "하지 않을 일"은 무엇인가?
4. 제품별로 끝까지 챙기는 역할이 명확한가?
5. 제품 간 자원 충돌은 어떻게 조정해야 하는가?

분과 입력 추가 문항:

- P1. 이번 반기 가장 중요한 제품/업무 영역 1~2개는 무엇이며, 그 이유는 무엇입니까?
- P2. 우선순위가 낮아졌거나 보류해야 할 일은 무엇입니까?
- P3. 제품별 책임 범위를 더 명확히 하기 위해 필요한 결정은 무엇입니까?

### 4.2 고객/시장/관련 팀 의견 분과

대상:

- PM, 프롬프트, 영업/서비스/마케팅 의견을 자주 다루는 구성원
- 고객 신호에 관심 있는 엔지니어 참여 가능

핵심 질문:

1. 도서관 구매자와 실제 이용자 신호가 구분되어 들어오는가?
2. 영업/마케팅/서비스/콘텐츠 조직의 요청은 어떤 기준으로 정식 서비스 논의로 넘어가는가?
3. 고객 요청과 제품 전략이 충돌할 때 누가 어떻게 결정하는가?
4. 사용자 데이터와 현장 의견 중 무엇이 부족한가?
5. 고객 의견이 팀에 전달되는 속도와 형식은 적절한가?

분과 입력 추가 문항:

- CUST1. 지금 제품 판단에 가장 부족한 고객/사용자 신호는 무엇입니까?
- CUST2. 내부 협업팀에서 들어오는 요청을 정리하기 위한 가장 작은 개선은 무엇입니까?
- CUST3. 고객 의견을 제품 우선순위로 바꾸는 기준을 한 문장으로 쓴다면 무엇입니까?

### 4.3 PM/프롬프트/요구사항 분과

대상:

- PM, 프롬프트 엔지니어
- 요구사항 작성/검토에 자주 관여하는 AI/BE/FE 구성원

핵심 질문:

1. 요구사항은 사용자 문제, 성공 기준, 제약을 충분히 포함하는가?
2. AI 기능 특유의 불확실성이 요구사항에 반영되는가?
3. PM/프롬프트/AI/BE/FE/디자인이 어느 시점에 함께 봐야 하는가?
4. 변경사항은 어떤 방식으로 공유되어야 하는가?
5. 완료 기준은 어디까지 포함해야 하는가?

분과 입력 추가 문항:

- REQ1. 현재 요구사항에서 가장 자주 빠지는 정보는 무엇입니까?
- REQ2. 요구사항 템플릿에 반드시 추가해야 할 항목 3개는 무엇입니까?
- REQ3. PM/프롬프트와 엔지니어 사이의 재작업을 줄이기 위한 4주 개선은 무엇입니까?

### 4.4 개발/운영 분과

대상:

- AI 엔지니어, 백엔드, 프론트엔드
- 인프라/운영 협업을 자주 다루는 구성원

핵심 질문:

1. 개발과 배포를 막는 가장 큰 지점은 무엇인가?
2. API, 데이터, 에러, 로딩, 권한, 예외 케이스 정의는 충분한가?
3. 인프라/운영 제약은 충분히 이른 시점에 드러나는가?
4. 로그, 모니터링, 재현 정보는 충분한가?
5. AI 품질/RAG 세부 기준은 분과 안에서 별도로 정리할 필요가 있는가?

분과 입력 추가 문항:

- ENG1. 개발/운영 흐름에서 가장 많이 기다리는 지점은 어디입니까?
- ENG2. 프론트엔드/백엔드/AI 사이 합의를 명확히 하기 위한 최소 문서 또는 체크리스트는 무엇입니까?
- ENG3. AI 품질/RAG 관련 논의가 필요하다면 공통 설문이 아니라 어떤 운영 장치로 다루는 것이 적절합니까?

### 4.5 CBT(사전 테스트)/신규 실험 분과

대상:

- AI Editor 등 CBT(사전 테스트)/신규 실험에 관여하는 구성원
- 신규 실험의 정식 서비스 기준에 관심 있는 구성원

핵심 질문:

1. AI Editor의 현재 목표는 학습, 출시, 검증, 판매 준비 중 어디에 가까운가?
2. CBT 단계에서는 어떤 수준의 공유와 검증이 필요한가?
3. 바이브코딩 등 빠른 구현 방식은 어떤 조건에서 유효한가?
4. CBT에서 정식 제품으로 넘어가는 기준은 무엇인가?
5. 소규모 전담 실험이 팀 전체의 전략과 어떻게 연결되어야 하는가?

분과 입력 추가 문항:

- CBT1. AI Editor에서 현재 가장 검증해야 할 가설은 무엇입니까?
- CBT2. CBT 단계에서 반드시 공유해야 하는 정보와 공유하지 않아도 되는 정보는 무엇입니까?
- CBT3. AI Editor 또는 신규 실험을 정식 제품 후보로 판단하기 위한 기준은 무엇입니까?

## 5. 분과 결과 통합 방식

각 분과 결과는 아래 기준으로 통합한다.

1. 겹치는 막힘은 하나로 묶는다.
2. 서로 충돌하는 제안은 의견 차이로 남긴다.
3. 실행 비용이 큰 제안과 4주 개선을 분리한다.
4. 최종적으로 2~3개의 4주 개선만 선택한다.
5. 선택되지 않은 제안은 백로그로 남긴다.

## 6. 최종 워크샵 산출물

워크샵 종료 시 남겨야 할 산출물은 다음이다.

- 4주 개선 2~3개
- 각 개선을 끝까지 챙기는 역할
- 개선 성공 기준
- 리뷰 날짜
- 보류한 중요 이슈 목록
- 다음 설문 또는 회고에서 다시 볼 질문
- 월간 펄스에서 확인할 문항 후보 5개 이하

## 7. 월간 펄스 연결

분과별 토론에서 선택된 4주 개선은 `02_survey_design/08_monthly_pulse_followup_pool.md`의 문항과 연결한다.

예시:

| 분과에서 선택한 개선 | 연결할 펄스 문항 |
|---|---|
| 제품별 끝까지 챙기는 역할과 우선순위 원칙 정리 | P-OWN01, P-OWN02, P-OWN03 |
| 요구사항 템플릿 개선 | P-REQ01, P-REQ02, P-REQ03 |
| 타팀 협업 요청 방식 정리 | P-EXT01, P-EXT02, P-EXT03 |
| CBT(사전 테스트)/신규 실험 기준 정리 | P-CBT01, P-CBT02, P-CBT03 |
| 새 구성원 적응/문서화 개선 | P-KNOW01, P-KNOW02, P-KNOW03 |
