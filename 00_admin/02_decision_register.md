# 설문 설계 결정 등록부

- 최초 작성일자: 2026-05-12
- 업데이트일자: 2026-05-12
- 업데이트 내용: v1.1 팀원 중심 응답 대상 14명 확정, 리드/기타 역할 라우팅 제거, 어려운 용어 제거 원칙 추가.
- 작성자: Codex

## 0. 판단 과정 요약

1. 설문 목적을 먼저 고정했다. 목적이 흔들리면 문항이 과도하게 늘고, 조직 진단과 개인 평가와 기술 감사가 섞이기 때문이다.
2. 현재 조직 맥락과 사용자 피드백을 우선 기준으로 두었다. 다른 세션 버전의 장점은 흡수하되, 최신 맥락과 충돌하는 문항은 배제했다.
3. 소규모 조직의 익명성 위험을 가장 큰 제약으로 보았다. 직군별 인원이 1~4명인 상태에서는 문항 설계보다 결과 공개 방식이 더 중요할 수 있다.
4. 최종 구현 기준은 `02_survey_design/07_final_question_bank_source_of_truth.md` 하나로 통일했다. 이전 문항집은 리서치/초안 기록으로 남긴다.

## 1. 확정 결정

| ID | 결정 | 근거 | 상태 |
|---|---|---|---|

| D03 | 팀장과 스쿼드 리드 관점은 필요 시 사전 인터뷰, 셀프 체크, 진행자 메모로 분리한다. | 리더 관점은 필요하지만 팀원 익명 설문과 같은 분석 단위가 아니다. | 권장 |
| D04 | AI Editor를 제품/워크스트림 선택지와 CBT/신규 실험 문항에 포함한다. | 현재 CBT 중인 서비스이며, 제품 포트폴리오와 실행 방식 논의에서 빠지면 최신성이 떨어진다. | 확정 |
| D05 | DBpia/KRpia/Citeasy는 실제 설문 문항에서 최소화한다. | 사업 맥락상 중요하지만, 이번 설문의 범위는 AI 사업부 운영체계다. | 확정 |
| D06 | AI/RAG 신뢰성 세부 진단은 공통 설문에서 제외한다. | 특정 직군 중심 기술 감사로 인식될 수 있고 사용자 피드백과도 맞지 않는다. | 확정 |
| D07 | AI/RAG 관련 주제는 엔지니어링/운영 분과 또는 별도 기술 회고에서 다룬다. | 중요성을 부정하지 않되, 공통 조직 설문의 목적과 분리한다. | 확정 |
| D08 | PM/리드 개인 효과성 평점 문항은 제외한다. | 워크샵 설문이 개인 평가로 해석되면 응답 솔직성이 낮아질 수 있다. | 확정 |
| D09 | 제품별 책임 역할표는 개인 평가가 아니라 분과 토론 전 사전 작성 자료로 사용한다. | 제품별 책임 구조를 명확히 하되, 개인별 점수화를 피한다. | 확정 |
| D10 | 모든 척도 문항은 높은 점수가 건강한 상태를 의미하도록 한다. | 해석과 분석 오류를 줄인다. | 확정 |
| D11 | 역문항은 사용하지 않는다. | 한국어 내부 설문에서 응답 혼동과 역코딩 오류 위험이 있다. | 확정 |
| D12 | 척도 문항은 `scale5na`를 사용한다. | 직접 경험하지 않은 문항에 억지 응답을 강요하면 평균이 왜곡된다. | 확정 |
| D13 | 역할별 문항은 라우팅으로 노출하되, 직군별 평균은 공개하지 않는다. | 역할별 병목은 필요하지만 n이 작아 식별 위험이 크다. | 확정 |
| D14 | 주관식 원문은 그대로 공유하지 않는다. | 개인 식별 가능 표현이나 특정 협업자 암시를 제거해야 한다. | 확정 |
| D15 | 워크샵의 최종 산출물은 4주 실험 2~3개로 제한한다. | 많은 개선안을 동시에 실행하면 동시에 진행 중인 일이 늘고 피로도가 증가한다. | 확정 |
| D16 | 워크샵 이후 월간 펄스를 운영한다. | 일회성 설문으로 끝나지 않고 실제 변화 여부를 추적해야 한다. | 권장 확정 |
| D17 | 배포 전 2~3명 대상 사전 인지 검증을 실시한다. | 도움말, 라우팅, 문항 해석의 오류를 사전에 줄인다. | 확정 |
| D18 | 역할 선택지는 제품/PM/프롬프트, AI 엔지니어링, 웹 엔지니어링(FE/BE) 3개로 제한한다. | 실제 응답 대상 14명의 역할을 모두 포괄하면서도 리드 1명이나 기타 응답으로 인한 식별 위험을 줄인다. | 확정 |
| D19 | 응답자에게 보이는 문항과 선택지에는 DRI, WIP, trade-off 같은 내부/영어식 용어를 쓰지 않는다. | 설문은 조직개발 용어 지식이 아니라 실제 업무 경험을 묻는 도구이므로, 자연스러운 한국어로 이해되어야 한다. | 확정 |

## 3. 비결정 사항

다음은 이번 설문 설계에서 의도적으로 결정하지 않는다.

1. AI 제품별 최종 우선순위.
2. PM 또는 리드별 성과 평가.
3. RAG 평가 체계의 구체 지표.
4. 조직 개편 여부.
5. 개인별 업무 재배치.

이 항목들은 설문 결과와 워크샵 토론을 통해 별도 의사결정 절차로 다루는 것이 맞다.
