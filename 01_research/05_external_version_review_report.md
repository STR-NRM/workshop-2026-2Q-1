## 2. 총평

다른 세션 버전은 "설문 문항 초안"이라기보다 "조직 진단 설문을 만들기 위한 리서치 패키지"에 가깝다. 이 점은 강점이다. 문서 간 역할이 분리되어 있고, 최종 문항집을 source of truth로 지정했으며, 이전 문서가 어떤 맥락에서 superseded 되었는지도 비교적 분명하게 남겼다.

다만 실제 워크샵 설문으로 그대로 채택하기에는 위험이 있다. 설문 목적이 조직 건강도 진단, AI 제품 전략 점검, ML/RAG 기술 운영 진단, PM/리드 평가, 월간 펄스 설계까지 넓게 확장되어 있다. 이 많은 목적을 한 설문 체계 안에 넣으면 응답자가 "조직 설문"으로 받아들이기보다 "평가/감사/전략 심문"으로 느낄 수 있다.

따라서 이 버전은 문항을 그대로 복사하기보다, 다음 세 가지 용도로 활용하는 것이 합리적이다.

1. 문서 운영 방식의 참고 자료
2. 설문 후속 운영과 의사결정 체크리스트의 참고 자료
3. 일부 선택 모듈과 분과 토론 주제의 후보군

현재 `workshop-2026-2Q-1`의 v2 설계는 최신 맥락 반영 측면에서 더 적합하다. 특히 AI Editor CBT, 직군별 라우팅, 질문 설명문, AI/RAG 공통 섹션 제외, 분과별 토론 구조가 이미 반영되어 있다. 반대로 다른 세션 버전은 리서치 추적성, 운영 로드맵, 월간 펄스, 의사결정 체크리스트 측면에서 배울 점이 많다.

## 4. 참고하면 좋은 점

### 4.1 Source of truth 문서 운영

가장 배울 만한 점은 최종 통합 문항집을 명확히 지정한 것이다. 다른 세션 버전은 `docs/09_final_consolidated_question_bank.md`를 최종 기준 문서로 두고, 이전 문서들은 근거와 변경 이력으로 활용한다.

우리도 최종 컨펌 단계에서는 다음과 같은 문서를 별도로 두는 것이 좋다.

- `02_survey_design/07_final_question_bank_source_of_truth.md`

이 문서는 앱 구현 또는 Google Form/Typeform 제작 시 유일한 기준이 되어야 한다. 기존 v2 문항집, 분과 토론 모듈, 앱 이전 메모는 참고 문서로 남기되, 최종 구현자는 source of truth만 보고도 구현할 수 있어야 한다.

### 4.2 의사결정 체크리스트

다른 세션 버전의 최종 문항집에는 확정 전 결정해야 할 항목들이 정리되어 있다. 이 방식은 매우 유용하다. 설문은 문항만 정하면 끝나는 것이 아니라, 응답 범위, 익명성, 결과 공개 수준, 직군 라우팅, 후속 워크샵 방식까지 결정해야 하기 때문이다.

우리 버전에 추가하면 좋은 결정 항목은 다음과 같다.

| 결정 항목 | 권장 방향 |
|---|---|
| 응답 대상 | 실무 스쿼드 15명 기본. 팀장은 별도 셀프체크 또는 해석 보조로 분리. |
| 역할 라우팅 | 직군 선택 후 공통 문항 + 역할별 문항 노출. |
| 직군별 결과 공개 | n이 작은 직군 평균은 공개하지 않음. |
| AI/RAG 세부 진단 | 공통 설문이 아니라 엔지니어링/운영 분과 또는 별도 기술 회고로 분리. |
| AI Editor | CBT/신규 실험 분과에서 별도 논의. 공통 문항에는 성숙도 차이와 공유 수준만 반영. |
| 주관식 원문 공개 | 개인 식별 가능성이 있으면 요약/재작성 후 공유. |
| 후속 실행 | 워크샵에서 4주 실험 2~3개만 확정. |

### 4.3 익명성 보호 원칙

다른 세션 버전은 소규모 조직에서 익명성을 지키는 원칙을 비교적 잘 정리했다.

참고할 원칙:

- 세부 직군별 평균을 그대로 공개하지 않는다.
- n이 작은 그룹은 묶어서 해석한다.
- 신규 입사자/합류자 경험은 별도 링크나 별도 해석으로 다루되, 개인 식별이 되지 않게 한다.
- 자유응답 원문은 필요한 경우 paraphrase한다.
- 특정 개인에 대한 코멘트로 보이는 응답은 조직 운영 이슈로 재분류한다.

이 원칙은 우리 설문에서 반드시 유지해야 한다. 현재 팀 규모에서는 직군별로 1~4명인 그룹이 많기 때문에, "직군별 인사이트"와 "개인 식별 위험"이 거의 같은 방향으로 움직인다.

### 4.4 월간 펄스 운영

다른 세션 버전의 M5 월간 펄스 아이디어는 가치가 크다. 워크샵 설문은 한 번의 진단으로 끝나면 행동 변화로 이어지기 어렵다. 5문항 정도의 짧은 월간 펄스를 만들면 워크샵에서 정한 운영 실험이 실제로 개선되는지 확인할 수 있다.

다만 월간 펄스는 처음부터 앱 구현 범위에 넣기보다, 워크샵 이후 후속 운영 문서로 분리하는 것이 좋다.

권장 방식:

- 워크샵에서 4주 실험을 2~3개만 확정한다.
- 실험별 성공 기준을 정한다.
- 4주 뒤 5문항 이하의 펄스로 변화 여부를 확인한다.
- 펄스 결과는 점수보다 "다음 조정"에 사용한다.

### 4.5 PM-제품 오너십 매트릭스

다른 세션 버전의 PM/SL 효과성 문항은 그대로 쓰기 어렵지만, PM-제품 매트릭스 아이디어는 유용하다.

특히 다음과 같은 형태는 분과 토론 전 pre-work로 적합하다.

| 제품/워크스트림 | DRI | Backup | Contributor | Consulted |
|---|---|---|---|---|
| AI Agent |  |  |  |  |
| AI Viewer |  |  |  |  |
| AI Idea |  |  |  |  |
| AI Reader |  |  |  |  |
| AI Editor |  |  |  |  |
| 공통 검색/AI 플랫폼 |  |  |  |  |

이 매트릭스는 개인 평가가 아니라 오너십 명확화 도구로 써야 한다. PM 또는 리드별 점수를 묻는 방식은 워크샵 설문 목적과 충돌할 수 있다.

### 4.6 사전 인지 검증

다른 세션 버전의 구현 로드맵에는 2~3명 대상 사전 테스트가 포함되어 있다. 이는 반드시 채택할 만하다.

사전 인지 검증에서 확인할 항목:

- 질문의 의미가 직군별로 동일하게 이해되는가?
- 설명문이 응답 방향을 유도하지 않는가?
- 직군 라우팅이 어색하지 않은가?
- 30분 안에 응답 가능한가?
- 특정 개인이나 팀을 떠올리게 하는 표현이 있는가?
- 주관식이 너무 많아 피로하지 않은가?

### 4.7 미검증 가정 명시

다른 세션 버전은 일부 문서에서 "unverified" 영역을 남겨둔다. 이 방식은 좋다. 특히 이번 설문은 조직 내부 정보를 바탕으로 만들기 때문에, 정확하지 않을 수 있는 전제를 명시해야 한다.

우리 최종 문서에도 다음 항목을 남기는 것이 좋다.

- AI 사업부 16명, 실무 스쿼드 15명 전제
- 팀장 응답 포함 여부
- AI Editor의 CBT 범위와 실제 담당 구조
- 외주/아이와즈 구성원의 응답 포함 여부
- 분과 토론 참여 단위
- 결과 공개 범위

## 5. 주의하거나 그대로 가져오면 안 되는 점

### 5.1 설문 목적이 너무 많이 섞여 있다

다른 세션 버전은 다음 목적을 한 체계에 동시에 넣고 있다.

- 조직 건강도 진단
- 신규 합류자 온보딩 진단
- AI/ML 기술 부채 진단
- RAG/AI 품질 체계 진단
- PM/스쿼드리드 효과성 평가
- 학술 출판 AI 전략 점검
- 월간 펄스 설계

각각은 의미가 있지만, 한 번의 워크샵 설문 안에 모두 들어가면 응답자가 설문의 목적을 명확히 이해하기 어렵다. 특히 PM/리드 효과성 평가는 조직 설문과 섞이면 방어적 응답을 유발할 수 있다.

우리 설계에서는 목적을 다음처럼 분리하는 것이 더 안전하다.

| 목적 | 처리 방식 |
|---|---|
| 조직 운영 건강도 | 공통 설문 |
| 직군별 협업 병목 | 역할별 라우팅 |
| 제품/우선순위/오너십 | 분과 토론 |
| AI/RAG 품질 체계 | 엔지니어링/운영 분과 또는 별도 기술 회고 |
| PM/리드 피드백 | 필요 시 별도 360/리더십 피드백으로 분리 |
| 후속 변화 추적 | 월간 펄스 |

### 5.2 특정 인명과 팀명을 문항에 직접 넣는 방식은 위험하다

다른 세션 버전에는 특정 협업자나 특정 팀을 직접 언급하는 문항이 있다. 내부 맥락 이해에는 도움이 되지만, 실제 설문 문항으로는 위험하다.

위험 요인:

- 응답자가 특정 개인 평가로 받아들일 수 있다.
- 결과 공유 시 정치적 해석이 생길 수 있다.
- 익명성이 약해질 수 있다.
- 협업 구조 문제가 개인 책임 문제로 바뀔 수 있다.

우리 설문에서는 "인프라 담당자", "디자이너", "서비스팀" 같은 표현도 필요 최소한으로 쓰고, 가능한 한 협업 인터페이스나 정보 흐름으로 바꾸는 것이 좋다.

예시:

- 부적합: "김주석/정연태와의 인프라 협업은 충분히 원활하다."
- 적합: "인프라/운영 제약은 일정 후반이 아니라 충분히 이른 시점에 확인된다."

### 5.3 AI/RAG 기술 문항을 공통 설문에 넣는 것은 현재 요구와 맞지 않다

다른 세션 버전은 RAGAS, 평가 데이터셋, 검색 품질, hallucination 관리 등 기술적 질문을 공통 설문에 가깝게 포함한다. 이 방향은 AI/ML 조직 진단으로는 의미가 있지만, 현재 사용자 피드백과는 맞지 않는다.

현재 요구는 AI/RAG 신뢰성 섹션을 굳이 공통 설문에 두지 않는 것이다. 따라서 이 주제는 다음처럼 처리하는 편이 낫다.

- 공통 설문: "AI 기능의 불확실성과 기술 제약이 요구사항에 반영되는가" 수준
- 엔지니어링/운영 분과: AI 품질 기준, 평가 루프, 운영 모니터링 논의
- 별도 기술 회고: RAG 평가, 검색 품질, hallucination, 데이터셋 관리

### 5.4 PM/SL 효과성 문항은 별도 절차가 필요하다

다른 세션 버전의 M4는 PM과 스쿼드 리드의 효과성을 묻는다. 이 아이디어는 문제의식 자체는 타당하지만, 워크샵 설문에 넣으면 위험하다.

위험 요인:

- 작은 조직에서 응답자가 누구를 평가했는지 추정될 수 있다.
- PM/리드 평가로 받아들여져 설문 전체의 신뢰도가 낮아질 수 있다.
- 조직 운영 개선보다 개인 성과 평가로 해석될 수 있다.
- 리더와 구성원의 권력 관계가 응답 솔직성을 낮출 수 있다.

대안은 다음과 같다.

- 워크샵 설문에서는 "의사결정자와 오너십이 명확한가"만 묻는다.
- PM/리드 피드백이 필요하면 별도 360 피드백으로 분리한다.
- PM별 평점보다 제품별 오너십 매트릭스를 사용한다.

### 5.5 강한 프레이밍은 설문 설계에는 부적합하다

다른 세션 버전에는 문서 설명에서 강한 판단 표현이 종종 나온다. 예를 들어 "Spotify에서도 안 쓰는 도구", "흡수합병 위험", "First Mover 우위", "10배 격차" 같은 표현은 읽는 사람에게 특정 결론을 먼저 떠올리게 할 수 있다.

전략 문서에서는 이런 표현이 논점을 선명하게 만들 수 있다. 그러나 설문 설계 문서나 응답자에게 보이는 문항에는 부적합하다. 설문은 응답자의 실제 체감을 수집해야 하며, 조사자의 진단을 응답자에게 주입하면 안 된다.

우리 문서에서는 다음 기준을 유지하는 것이 좋다.

- 응답자에게 보이는 문장에는 위기감 조성 표현을 넣지 않는다.
- 특정 프레임워크를 비판하는 표현보다 현재 팀의 필요에 맞는지로 설명한다.
- 전략적 위협은 분과 토론 참고자료로만 다룬다.

### 5.6 역문항은 신중해야 한다

다른 세션 버전은 Edmondson 심리적 안전감 문항 등에서 역문항을 사용한다. 학술적 원형을 보존한다는 장점은 있다. 하지만 한국어 조직 설문, 짧은 응답 시간, 소규모 내부 설문에서는 해석 오류 위험이 있다.

우리 v2는 이미 "점수가 높을수록 건강한 상태"로 통일하고 역문항을 쓰지 않는 원칙을 세웠다. 이 원칙은 유지하는 것이 좋다.

역문항이 꼭 필요하다면 다음 조건을 충족해야 한다.

- 질문 아래에 해석 설명을 붙이지 않아도 오해가 없어야 한다.
- 분석자가 역코딩 실수를 하지 않도록 구현에서 명확히 처리해야 한다.
- 응답자가 부정문을 두 번 해석해야 하는 문장은 피해야 한다.

현재 상황에서는 역문항을 쓰지 않는 쪽이 더 단순하고 안전하다.

### 5.7 산업 리서치의 양이 설문 적합성을 보장하지 않는다

다른 세션 버전은 학술 출판 AI 시장, 글로벌 경쟁사, B2B2C 구조, COUNTER 등 산업 맥락을 많이 다룬다. 이 리서치는 유용하다. 그러나 조직 설문에 필요한 것은 "모든 전략 맥락"이 아니라 "구성원이 일하는 방식에 영향을 주는 맥락"이다.

따라서 산업 리서치는 다음처럼 좁혀 쓰는 것이 좋다.

- 고객 신호가 제품 결정에 반영되는가
- 구매자와 실제 이용자 신호가 구분되는가
- 제품별 성공 기준이 정의되어 있는가
- 시장/영업/서비스 피드백이 우선순위로 번역되는가

경쟁사 전략이나 시장 수치는 응답자 설문 문항이 아니라 워크샵 배경자료 또는 리더십 전략 토론 자료에 더 적합하다.

## 8. 최종 권고

다른 세션 버전은 "깊게 조사한 연구 노트"로는 가치가 높다. 특히 문서 체계, source of truth 운영, 익명성 보호 원칙, 월간 펄스, 사전 테스트, 구현 로드맵은 우리 버전에 적극 반영할 만하다.

그러나 "실제 이번 상반기 워크샵 설문"으로는 그대로 사용하지 않는 편이 낫다. 이유는 명확하다. 현재 필요한 설문은 확장된 15인 실무 스쿼드가 제품/역할/협업/의사결정/실험 운영을 어떻게 개선할지 찾는 도구다. 다른 세션 버전은 여기에 AI 기술 감사, PM/리드 효과성 평가, 산업 전략 진단까지 한꺼번에 넣으려는 경향이 있다.

따라서 최종 설계 방향은 다음이 가장 합리적이다.

1. 문항의 기본 골격은 현재 `2Q-1` v2를 유지한다.
2. 다른 세션 버전에서는 문항보다 운영 장치를 가져온다.
3. 최종 컨펌 전 source of truth 문항집과 결정 체크리스트를 만든다.
4. PM/리드 평가, AI/RAG 기술 평가는 이번 공통 설문에서 분리한다.
5. 분과 토론에는 PM-제품 오너십 매트릭스와 월간 펄스 후속 운영을 추가한다.
6. 배포 전 2~3명 대상 사전 인지 검증을 반드시 한다.

한 문장으로 요약하면, 다른 세션 버전은 "참고할 만한 깊이"는 크지만 "그대로 배포할 문항 안정성"은 낮다. 이번 작업에서는 그 버전의 리서치 운영법과 후속 관리 장치를 흡수하고, 실제 설문 문항은 현재 v2처럼 더 중립적이고 목적이 분명한 구조로 유지하는 것이 좋다.
