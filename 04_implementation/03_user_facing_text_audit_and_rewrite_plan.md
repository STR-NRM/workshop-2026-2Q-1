## 0. 판단 과정 요약

1. 사용자 화면 텍스트는 `src/data/questions.js`, `src/pages/Result.jsx`, `src/utils/openaiAnalysis.js`, `src/pages/Landing.jsx`, `src/pages/Survey.jsx`, `src/pages/Complete.jsx`, `src/components/survey/QuestionRenderer.jsx`, `src/components/common/ProgressBar.jsx`, `src/App.jsx`를 기준으로 스캔했다.
2. 판단 기준은 "AI 사업부 내부에 오래 있던 사람만 아는 말인가", "신규 PM/FE/BE도 바로 이해하는가", "응답자가 무엇을 기준으로 답해야 하는가", "결과 화면에서 숫자가 결론처럼 보이지 않는가"로 잡았다.
3. 외부 기준은 18F Plain Language, Digital.gov Plain Language, Pew Research 설문 문항 작성 가이드, Nielsen Norman Group 사용성 휴리스틱을 참고했다.
4. 내부 도메인상 필요한 약어와 제품명은 무조건 제거하지 않았다. 대신 첫 화면, 질문, 도움말, 결과 해석에서 가능한 쉬운 한국어를 함께 쓰는 방향으로 정리한다.
5. 설문 문항은 의미가 바뀌면 안 되므로, 지표 구조와 문항 ID는 유지하고 제목/본문/도움말의 이해 난이도만 낮춘다.

## 1. 참고 기준

- 18F Content Guide: 쉬운 말, 짧은 단어, 내부 전문어와 비유적 표현을 줄이라는 원칙을 제시한다. https://guides.18f.org/content-guide/our-approach/plain-language/
- Digital.gov Plain Language: 독자가 알고 싶어 하는 내용부터 쓰고, 독자별로 필요한 내용을 구분하라고 제안한다. https://digital.gov/guides/plain-language/principles/write-for-reader
- Pew Research Center: 설문 문항은 명확하고 구체적이어야 하며, 한 번에 하나만 물어야 하고, 낯선 약어와 전문어를 피해야 한다고 정리한다. https://www.pewresearch.org/writing-survey-questions/
- Nielsen Norman Group: 화면은 사용자의 언어를 써야 하고 내부 용어를 피해야 하며, 오류 문구는 쉬운 말로 문제와 해결 방법을 알려야 한다. https://media.nngroup.com/media/articles/attachments/Heuristic_Summary1_Letter-compressed.pdf

## 2. 적용 원칙

- 내부어를 쉬운 말로 바꾼다: `병목`은 `막히는 지점`, `이견`은 `의견 차이`, `축`은 `영역`, `라우팅`은 `직무별 문항을 나누기`로 바꾼다.
- 전문어는 필요한 곳만 남긴다: `PM`, `FE`, `BE`, `AI`, `API`, `CBT`는 팀 업무상 필요하므로 유지하되, UI에서는 가능한 한국어 설명을 붙인다.
- 결과 화면은 결론처럼 말하지 않는다: `우선 신호`, `낮은 평균`, `분산`, `N/A`는 원인을 단정하지 않고 `먼저 확인할 후보`, `응답 차이`, `판단 어려움`으로 설명한다.
- 설문 도움말은 응답 기준을 알려주되 답을 유도하지 않는다: "좋다/나쁘다" 대신 "무엇을 기준으로 답할지"를 쓴다.
- AI 리포트 프롬프트는 팀원이 읽는 최종 보고서에 영향을 주므로, 어려운 컨설팅 용어와 영어식 표현을 줄이고 쉬운 한국어 출력을 강제한다.

## 4. 적용 범위

- 설문 응답 화면: 질문 제목, 질문 본문, 도움말, 선택지, 척도 라벨, placeholder, 진행/검증/오류 문구.
- 결과 화면: 요약 카드, 우선 신호, 문항별 해석, 서술형 안내, AI 리포트 탭, 리포트 생성 상태와 빈 화면.
- AI 리포트 프롬프트: 생성되는 보고서의 섹션명, 작성 원칙, 쉬운 말 규칙, 출력 용어.
- 기존 응답 데이터: 이미 저장된 예전 선택지 문구가 결과 화면에 다시 노출되지 않도록 집계 단계에서 현재 표현으로 정규화한다.
- 문서: 구현 TODO와 완료 리포트에 이번 문구 정리 내용을 반영.

## 5. 보류한 표현과 이유

- `AI`, `PM`, `FE`, `BE`, `API`, `CBT`는 팀 업무 자체와 연결된 약어라 완전히 제거하지 않는다. 다만 화면에서는 `웹 개발(프론트/백엔드)`, `CBT(사전 테스트)`처럼 가능한 설명을 붙인다.
- `Executive Summary`는 사용자가 명시적으로 유지하길 원한 리포트 구조라 유지한다.
- 제품명 `AI Agent`, `AI Viewer`, `AI Idea`, `AI Reader`, `AI Editor`는 조직 내 실제 제품명이므로 유지한다.
