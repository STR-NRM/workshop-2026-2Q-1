# 설문 구현 및 워크샵 실행 로드맵

- 최초 작성일자: 2026-05-12
- 업데이트일자: 2026-05-13
- 업데이트 내용: v1.3 GitHub Pages 정적 배포 기준 AI 분석 실행 방식과 종합/비주관식/주관식 리포트 분리 반영.
- 작성자: Codex
- 적용 대상: 2026년 상반기 AI 사업부 워크샵 준비 및 후속 운영

## 0. 판단 과정 요약

1. 설문 문항 확정만으로는 워크샵 품질이 보장되지 않는다. 구현, 익명성, 사전 검증, 분석, 분과 토론, 후속 펄스가 하나의 흐름으로 연결되어야 한다.
2. 기존 4Q 앱을 재사용하되, 데이터 경로와 프롬프트가 섞이면 결과 신뢰도가 떨어진다. 따라서 source of truth 기반 재구현과 데이터 분리를 필수 조건으로 둔다.
3. 소규모 조직에서는 배포 전 사전 인지 검증이 중요하다. 문항 하나가 특정 개인이나 직군을 떠올리게 하면 응답 품질이 크게 떨어질 수 있다.
4. 워크샵 후 실행을 위해 월간 펄스를 별도 운영 단계로 포함했다.

## 3. 구현 체크리스트

### 3.1 설문 앱/폼

- `scale5na`를 지원한다.
- 모든 문항의 도움말을 질문 아래에 표시한다.
- `META_ROLE`에 따라 역할별 모듈이 하나만 노출된다.
- `META_EXTERNAL`에 따라 외부 협업 모듈이 조건부 노출된다.
- `META_CBT`에 따라 CBT/신규 실험 모듈이 조건부 노출된다.
- 주관식 입력에는 개인명/특정인 언급을 피하라는 안내가 보인다.
- 완료 화면에는 "결과는 개인 평가가 아니라 워크샵 운영 개선에 사용된다"는 문구를 넣는다.

### 3.2 데이터/보안

- 2025 4Q 데이터와 2026 2Q-1 데이터 경로를 분리한다.
- 가능하면 새 Firebase 프로젝트를 사용한다.
- 같은 Firebase를 쓰면 `/surveys/2026-2Q-1/...`처럼 버전 경로를 둔다.
- OpenAI API 키는 소스, GitHub Actions secret, Firebase에 저장하지 않는다.
- AI 분석은 결과 화면에서 진행자가 API key를 입력하고, 어느 AI 리포트 탭에서든 생성 버튼을 누르면 3종 익명 집계 데이터를 동시에 OpenAI Responses API에 보내 생성한 뒤 Firebase에 저장한다.
- 이 방식은 GitHub Pages 정적 배포와 Firebase Spark 플랜만으로 작동한다. 단, API key를 입력한 브라우저에서는 해당 요청에 key가 사용되므로 공개 단말이나 화면 공유 상황에서는 주의한다.
- 결과/AI 분석 페이지는 팀 공유용으로 공개 조회한다. 응답 저장은 Anonymous Auth로 제한한다.

### 3.3 AI 분석 프롬프트

AI 분석은 다음 원칙을 지켜야 한다.

- 개인 또는 특정 직군을 문제 원인으로 단정하지 않는다.
- 낮은 점수를 리더십 실패, 개발 역량 부족, PM 문제로 바로 해석하지 않는다.
- AI/RAG 품질 세부 지표를 공통 조직 설문 결과에서 과도하게 추론하지 않는다.
- 평균, 분산, N/A 비율, 주관식 반복 테마를 함께 본다.
- 출력은 Executive Summary로 시작하고, 전문가 검토형 보고서 수준의 구조와 근거를 갖추되 팀원이 읽기 쉬운 한국어로 작성한다.
- Executive Summary는 10단어 이내 한 문장 결론, 쉬운 분석 요약, 그래서 무엇을 해야 할지 3~5개 제안으로 시작한다.
- 세부 섹션은 10단어 이내 한문장 정리와 한문장 제안을 먼저 보여준 뒤 상세 근거를 이어간다.
- 결과 화면은 한 문장 결론, 한문장 정리, 한문장 제안을 배경이 있는 강조 박스로 표시한다.
- 강점, 병목, 이견, 4주 실험 후보, 중단 기준, 워크샵 토론 질문을 함께 제시한다.

## 4. 사전 인지 검증 프로토콜

대상:

- PM/프롬프트 1명
- 엔지니어링 1명
- 신규 또는 복합 역할에 가까운 구성원 1명

확인 질문:

1. 질문의 의미가 바로 이해되는가?
2. 도움말이 특정 답을 유도하지 않는가?
3. 특정 개인이나 팀을 떠올리게 하는 문항이 있는가?
4. 역할 라우팅이 자연스럽게 작동하는가?
5. N/A를 선택해야 하는 문항이 과도하게 많은가?
6. 30분 안에 응답 가능한가?

수정 기준:

- 2명 이상이 같은 문항을 오해하면 문항 또는 도움말을 수정한다.
- 특정 개인을 떠올리게 한다는 피드백이 있으면 표현을 구조/프로세스 중심으로 바꾼다.
- N/A가 많이 나올 것으로 예상되는 공통 문항은 라우팅 문항으로 이동하거나 삭제한다.

## 5. 워크샵 당일 운영

권장 흐름:

1. 설문 목적과 익명성 원칙 재확인
2. 전체 결과 요약 공유
3. 강점 3개, 병목 3~5개, 분산 큰 문항 확인
4. 제품별 책임 역할표 사전 작성 내용 공유
5. 분과별 토론
6. 4주 실험 후보 통합
7. 최종 2~3개 실험 선택
8. 책임 역할, 성공 기준, 리뷰 날짜 확정

금지:

- "누가 낮게 답했을까" 추정
- 특정 직군 방어 또는 책임 전가
- 설문 결과를 개인 평가처럼 사용
- 모든 문제를 한 번에 고치려는 결정

## 6. 워크샵 이후

### D+7 점검

- 실험이 실제로 시작되었는지 확인한다.
- 책임 역할이 불명확하면 즉시 정리한다.
- 실행을 막는 다른 팀의 도움이 필요하면 별도 이슈로 분리한다.

### D+28 월간 펄스

- `02_survey_design/08_monthly_pulse_followup_pool.md`에서 5문항 이하로 선택한다.
- 점수와 주관식 조정 의견을 함께 본다.
- 실험을 유지, 수정, 중단 중 하나로 결정한다.

## 7. 완료 기준

이번 워크샵 준비가 완료되었다고 보려면 다음이 충족되어야 한다.

1. source of truth 문항집이 사용자 최종 컨펌을 받았다.
2. 앱/폼이 source of truth와 일치한다.
3. 테스트 응답으로 데이터 저장, 라우팅, 결과 집계가 확인되었다.
4. 사전 인지 검증에서 치명적 오해가 제거되었다.
5. 워크샵 자료가 개인 식별 위험 없이 준비되었다.
6. 4주 실험 리뷰 날짜가 캘린더에 잡혔다.
